Sequence Diagram - Zero-Trust Architecture

Timeline of access flows and security controls on the Sentinel platform

1. Authentication and Edge

The privileged user accesses the platform via the gateway which orchestrates entry security.

  • Kong Gateway: Intercepts public traffic, enforces HTTPS, and applies rate-limiting.
  • Auth0 OIDC: Delegates authentication and enforces Multi-Factor Authentication (MFA).
  • RBAC Validation: The "PAM_Administrator" role is verified via JWT.

2. Zero-Trust Network

No internal traffic is implicitly trusted, the network is strictly segmented.

  • Istio Service Mesh: Injects sidecars to encrypt communications (mutual TLS).
  • Default-Deny: Rejects any traffic that does not expressly originate from the Kong gateway.
  • Isolation: Communication to the privileged VM is filtered by an IP allow-list.

3. Hardened Portal

The client application is designed to resist hostile browser attacks.

  • Non-root NGINX: Read-only container without Linux privileges.
  • Strict CSP: Prevents loading unsigned scripts (Subresource Integrity).
  • In-Memory Token: Tokens are not stored in LocalStorage to counter XSS attacks.

4. Secrets and Audits

The privileged core manages access ephemerally and traceably.

  • HashiCorp Vault: Generates temporary credentials on the fly (Dynamic Secrets).
  • Azure Key Vault: Securely escrows the Vault unseal keys.
  • Splunk SIEM: Ingests the audit flow in real time for anomaly detection.
