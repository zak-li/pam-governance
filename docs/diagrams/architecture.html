System Architecture - Zero-Trust PAM Platform

Components, trust zones and data flows of the PAM Governance platform

1. Edge and Identity

Entry to the platform is brokered by the gateway and delegated to a dedicated identity provider.

  • Kong Gateway: public LoadBalancer, HTTPS redirect, and rate-limiting at the edge.
  • Auth0 OIDC: SSO with enforced MFA; RBAC roles carried in the token.
  • Least-privilege default: users land as operator; admin needs a group claim.

2. Zero-Trust Network

No traffic is implicitly trusted; the cluster and the VM are strictly segmented.

  • Istio mesh: sidecars encrypt pod-to-pod traffic with mutual TLS.
  • NetworkPolicy: ingress limited to the Kong gateway and the mesh.
  • NSG allow-list: the privileged VM is reachable only from the admin IP.

3. Hardened Portal

The Angular app is built into a registry image and served to resist a hostile browser.

  • Angular + ACR: image built by az acr build, pulled via AcrPull.
  • Non-root nginx: read-only filesystem, dropped Linux capabilities.
  • Strict CSP: in-memory tokens, no local storage (XSS resistant).

4. Secrets and Audit

The privileged core issues access ephemerally and records everything.

  • HashiCorp Vault: dynamic secrets (SSH CA, database, transit, KV).
  • Azure Key Vault: escrows the Vault unseal keys and root token.
  • Splunk SIEM: ingests the Vault audit device into the pam_audit index.
